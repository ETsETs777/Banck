# Безопасность Spektors

Краткий аудит и рекомендации для продакшена. Код ориентирован на локальную разработку и Docker; перед выводом в интернет закройте описанные риски.

## Что уже сделано в коде

- **CORS** — только origin’ы из `config/apps.yaml`, не `*`. С `allow_credentials=True` это важно: произвольный сайт не шлёт запросы с куками к API. В `Access-Control-Expose-Headers` проброшены `X-Session-Id`, `X-Thread-Id`, **`X-Request-ID`** (корреляция с `@spektors/api-client`).
- **Публичное API** — заголовок **`X-App-Id`** привязан к реестру приложений; неизвестный id → 403.
- **`POST /api/v1/chat/completions`** — тоже требует **`X-App-Id`**; модель берётся из конфига приложения, тело не может переключить модель (защита от злоупотребления Ollama).
- **Internal** — при заданных `INTERNAL_ADMIN_TOKEN` / `INTERNAL_DEV_TOKEN` нужен `Authorization: Bearer …`. Сравнение токена — через хэш и `hmac.compare_digest` (без простого `==`).
- **`INTERNAL_AUTH_STRICT=1`** — если для admin/dev токен в env **не задан**, соответствующие маршруты отвечают **503** `internal_auth_not_configured` (закрыть «открытый internal» в проде).
- **Rate limit** — для тяжёлых POST (чат, RAG, completions) см. `RATE_LIMIT_PER_MINUTE`; ответ **429** с заголовком **`Retry-After`**.
- **SQL** — запросы через asyncpg с плейсхолдерами, конкатенация пользовательского ввода в SQL отсутствует.
- **Валидация** — длина полей чата/RAG в Pydantic; `session_id` / `thread_id` в теле чата и в пути — hex, ограниченная длина.
- **OpenAPI** — в проде можно отключить: переменная окружения **`DISABLE_OPENAPI=1`** (или `true` / `yes`) скрывает `/docs`, `/redoc`, `/openapi.json`.

## Риски и рекомендации

### Internal API без токенов

Если **`INTERNAL_*_TOKEN` не заданы**, маршруты `/internal/*` **доступны без авторизации**. Это удобно только для локальной разработки. В любом общедоступном окружении **обязательно** задайте длинные случайные токены.

### Админка (web-admin) и Bearer в браузере

Страница «Очередь чатов» передаёт токен из поля формы в `Authorization`. Токен живёт в памяти вкладки (не в `localStorage`), но при **XSS** на origin админки его могут перехватить. Для продакшена предпочтительно: **серверный BFF** (Next Route Handler / отдельный бэкенд), который держит токен только на сервере, а браузер ходит к своему origin без секрета.

### Docker Compose по умолчанию

- **Postgres** — пароль `spektors` в открытом виде; порт **5432** проброшен на хост. Для прода: сильный пароль, не публиковать 5432 наружу, при необходимости TLS.
- **Ollama (11434)** и **Chroma (8001)** — без встроенной авторизации в этом стеке. Не выставляйте их в интернет; доступ только из приватной сети / VPN / SSH-туннель.
- Закомментированные **`INTERNAL_*_TOKEN`** в `docker-compose.yml` нужно задать перед боевым использованием.

### Публичные health и meta

- **`/api/v1/health/*`** и **`/api/v1/meta/config-check`** раскрывают состояние зависимостей и список `app_id` — удобно для мониторинга, но даёт разведку. При необходимости ограничьте доступ на уровне reverse proxy или вынесите за VPN.

### Rate limiting только в приложении

В API есть лимит на частоту тяжёлых POST (см. `RATE_LIMIT_PER_MINUTE`). Для публичного интернета дополнительно имеет смысл лимит на reverse proxy (nginx, Traefik, Cloudflare).

### Секреты в репозитории

Не коммитьте `.env` с реальными токенами (в `.gitignore` уже есть `.env` / `.env.local`).

## Чеклист перед продом

1. Задать **`INTERNAL_ADMIN_TOKEN`** и **`INTERNAL_DEV_TOKEN`** (или отключить/не публиковать internal-маршруты); в проде включить **`INTERNAL_AUTH_STRICT=1`**.
2. Сменить пароль БД и убрать лишние published ports у Postgres/Ollama/Chroma.
3. Проверить **`cors_origins`** в `config/apps.yaml` — только реальные HTTPS-origin’ы фронтов.
4. Включить **`DISABLE_OPENAPI=1`**, если Swagger не нужен снаружи.
5. Рассмотреть **BFF** для операторской панели вместо токена в браузере.
6. Включить **TLS** на границе (reverse proxy) и заголовки вроде `X-Content-Type-Options: nosniff`.
