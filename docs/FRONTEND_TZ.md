# ТЗ: фронтенд Spektors (веб-банк + LLM + админка)

Исходного единого файла ТЗ в репозитории нет; этот документ фиксирует целевую модель UI, ролей и интеграций. Версия 0.2, март 2026.

## 1. Назначение приложений

| Приложение      | Порт | Пользователь        | Назначение |
|-----------------|------|---------------------|------------|
| **web-client**  | 3000 | Клиент банка        | Основной кабинет, чат с AI (и с участием оператора в фоне). |
| **web-lite**    | 3001 | Клиент (lite)       | Упрощённый UI, тот же API с `app_id: web_lite`. |
| **web-admin**   | 3002 | Оператор / админ    | Очередь диалогов, ответ от лица человека, мониторинг, настройки (позже). |
| **web-dev**     | 3003 | Разработчик         | Отладка API, логи, флаги (позже). |

Все публичные вызовы к API идут с заголовком **`X-App-Id`**, согласованным с `config/apps.yaml` и CORS.

## 2. Дизайн-система

- **Токены:** пакет `@spektors/ui-tokens` (`theme.css`, Tailwind preset): стеклянные панели (`--glass`, `--glass-border`), акцент, тёмная/светлая тема (переключатель — на всех приложениях, единое поведение).
- **Типографика:** DM Sans, иерархия: заголовок страницы → секция → подпись / метка.
- **Паттерны:** карточки с `rounded-2xl`, `backdrop-blur`, `shadow-glow`; первичная кнопка на акценте; состояния loading / error / пусто.
- **Доступность:** контраст текста, `aria` для переключателя языка и чата, фокус-кольца.

## 3. Интернационализация

- Локали **ru**, **en**, **nl**; пресет `@spektors/next-i18n` + словари в `@spektors/messages` для **`web-client`**, **`web-lite`**, **`web-admin`** (web-dev — статический каркас, i18n по мере роста UI).
- Любая новая пользовательская строка — сначала ключи в JSON, затем компонент.

## 4. Чат: совместная работа AI и человека

### 4.1 Модель данных (концепт)

- Сообщение в треде использует роли в формате OpenAI: `user` | `assistant` | при необходимости `system`.
- Дополнительно: **`msg_source`**: `user` | `model` | `human` | `system` — кто фактически сформировал ответ.
- Для ответов оператора: `role=assistant`, `msg_source=human`, **`author_label`** (отображаемое имя, например «Поддержка»).

Клиентский чат и LLM видят в истории ответы оператора как обычные `assistant` (контент как у модели); в UI можно показывать бейдж «Оператор» по `msg_source` + `author_label`.

### 4.2 Потоки

1. **Только AI:** пользователь пишет → API вызывает Ollama → сохраняется `assistant` + `msg_source=model`.
2. **Оператор подключается:** админ через защищённый internal API отправляет ответ → в тред попадает `assistant` + `msg_source=human` + `author_label`.
3. **Смешанный тред:** пользователь продолжает писать; очередной ответ снова может быть моделью или человеком (решение оператора / правил маршрутизации — позже).

### 4.3 Админка и безопасность

- Ответы оператора в v1: **`POST /internal/admin/v1/sessions/{session_id}/threads/{thread_id}/reply`** с **`Authorization: Bearer`** и переменной **`INTERNAL_ADMIN_TOKEN`** на API.
- **Не хранить** долгосрочно токен в `localStorage` продакшена; для dev допустимо поле ввода токена на странице «Очередь». Целевой вариант — серверный BFF (Next Route Handler) с секретом только на сервере.

### 4.4 UI web-client / web-lite (этапы)

1. Сейчас: общий `@spektors/chat-ui`, статусы БД/LLM, стрим.
2. Далее: подгрузка истории с бейджами по `msg_source` / `author_label`.
3. Далее: индикатор «печатает оператор» / уведомления (WebSocket или polling — отдельное ТЗ).

### 4.5 UI web-admin (этапы)

1. Каркас: боковая навигация, glass layout.
2. Страница «Очередь чатов»: ввод session/thread, токена (dev), текста ответа и подписи оператора → вызов internal reply.
3. Далее: список активных тредов из API (нужен read-only internal list — следующий инкремент).

## 5. Интеграция с API

- `@spektors/api-client`: health, chat, RAG, meta; для админских сценариев — методы с `kind: "admin"` и Bearer (по мере добавления роутов).
- Базовый URL: `NEXT_PUBLIC_API_URL` (по умолчанию `http://localhost:8000`).

## 6. web-dev

- Документация ссылками на Swagger (`/docs`), health-эндпоинты, копирование `app_id` — минимальный каркас; расширение по мере появления debug API.

## 7. Приоритеты внедрения (фронт)

1. Единый визуальный каркас + i18n там, где уже есть чат (client, lite).  
2. Отображение `msg_source` / `author_label` в `chat-ui`.  
3. Админ: список тредов + takeover.  
4. Тема light/dark с сохранением в `localStorage`.  
5. View Transitions (Next) для плавных переходов — опционально.

## 8. Нефункциональные требования

- Сборка без ошибок ESLint/TS в CI (когда появится).
- Не тянуть секреты админки в клиентский бандл в production.
